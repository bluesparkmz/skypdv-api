BlueSpark Identity Platform

Arquitetura de autenticação centralizada para os apps BlueSpark MZ

Esta documentação define como um app do ecossistema, como SkyVenda MZ, deve autenticar utilizadores usando a BlueSpark MZ como provedor central de identidade, assumindo backend em FastAPI.

Conta única Google OAuth central Backends sincronizados central_user_id FastAPI

Visão geral

Antes

Cada app tinha utilizadores, senhas e login social próprios.

Agora

A BlueSpark MZ gere identidade, login, social auth e tokens para todos os apps.

+--------------------+ +-------------------------+ | Frontend do App | | Frontend BlueSpark MZ | | Ex: SkyVenda MZ | | Login / Register | +---------+----------+ +------------+------------+ | | | login local ou google | | | v v +---------+------------------------------------+----------+ | BlueSpark MZ Accounts / Auth Backend | | - users globais | | - password login | | - google one tap / oauth | | - refresh token | | - introspect / validate token | +---------------------------+------------------------------+ | | token + dados do user v +---------------------------+------------------------------+ | Backend do App Consumidor | | - recebe token BlueSpark | | - valida token com BlueSpark | | - resolve user local por central_user_id | | - cria user local se faltar | | - continua regras de negocio do produto | +----------------------------------------------------------+

Princípio arquitetural

O utilizador existe primeiro na BlueSpark MZ. Cada produto mantém apenas o que é local ao seu domínio, mas nunca volta a ser dono da autenticação principal.

Um app consumidor pode ter tabela local users, mas essa tabela serve para contexto do produto. A identidade central continua a ser da BlueSpark.

Responsabilidades

Componente Responsabilidade
BlueSpark frontend Registo, login próprio, gestão de conta e recuperação de senha.
BlueSpark backend Emitir tokens, autenticar Google, gerir memberships por produto e validar sessões.
Frontend do app Iniciar login no contexto do produto e enviar token ao backend do próprio app.
Backend do app Validar o token BlueSpark, criar ou atualizar user local e aplicar regras do produto.

Stack assumida

Backend central

FastAPI responsável por login, Google auth, emissão de token, refresh e introspecção.

Backend do app consumidor

FastAPI responsável por chamar a auth central, sincronizar o user local e proteger rotas do produto.

Fluxo recomendado para qualquer app

O utilizador tenta entrar no app consumidor, por exemplo SkyVenda MZ.
O app usa a BlueSpark MZ para autenticar com senha ou Google.
A BlueSpark MZ emite access_token e refresh_token associados ao product_code.
O backend do app valida o token na BlueSpark e procura o utilizador local por central_user_id.
Se não existir, o backend cria o utilizador local vinculado ao utilizador global.
A sessão continua no app com o contexto local já sincronizado.

Google Auth centralizado

Todos os apps devem usar o mesmo cliente Google da BlueSpark MZ. O utilizador pode autenticar no SkyVenda MZ sem abrir a BlueSpark manualmente.

Exemplo base em FastAPI

import os
import httpx

from fastapi import HTTPException, status

BLUESPARK_ACCOUNTS_URL = os.getenv("BLUESPARK_ACCOUNTS_URL", "https://accounts.bluesparkmz.com").rstrip("/")
BLUESPARK_PRODUCT_CODE = os.getenv("BLUESPARK_PRODUCT_CODE", "skyvenda")
BLUESPARK_AUTH_TIMEOUT_SECONDS = 15

def _accounts_url(path: str) -> str:
    return f"{BLUESPARK_ACCOUNTS_URL}{path}"

def _accounts_request(method: str, path: str, json: dict | None = None, token: str | None = None) -> dict:
    headers = {}
    if token:
        headers["Authorization"] = f"Bearer {token}"

    try:
        with httpx.Client(timeout=BLUESPARK_AUTH_TIMEOUT_SECONDS) as client:
            response = client.request(method, _accounts_url(path), json=json, headers=headers)
    except httpx.HTTPError as exc:
        raise HTTPException(
            status_code=status.HTTP_502_BAD_GATEWAY,
            detail="BlueSpark auth indisponivel no momento."
        ) from exc

    data = response.json() if response.content else {}
    if response.status_code >= 400:
        raise HTTPException(
            status_code=response.status_code,
            detail=data.get("detail", "Falha na auth central.")
        )

    return data

Implementação FastAPI para resolver o user local

from fastapi import HTTPException, status
from sqlalchemy.orm import Session

def sync_local_user_from_claims(db: Session, claims: dict):
    central_user_id = str(claims.get("central_user_id") or claims.get("sub") or "").strip()
    email = (claims.get("email") or "").strip().lower()
    username = (claims.get("username") or email.split("@")[0] or f"user{central_user_id}").strip()
    full_name = (claims.get("name") or claims.get("full_name") or username).strip()

    if not central_user_id:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Token central sem utilizador.")

    if not email:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Token central sem email.")

    user = db.query(User).filter(User.central_user_id == central_user_id).first()
    if not user:
        user = db.query(User).filter(User.email == email).first()

    if user:
        user.central_user_id = central_user_id
        if not user.name:
            user.name = full_name
        db.commit()
        db.refresh(user)
        return user

    user = User(
        central_user_id=central_user_id,
        email=email,
        username=username,
        name=full_name,
    )
    db.add(user)
    db.commit()
    db.refresh(user)
    return user

Implementação FastAPI para proteger rotas

from fastapi import Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="user/token")

def introspect_central_token(token: str) -> dict:
    return _accounts_request(
        "POST",
        "/auth/introspect",
        json={
            "token": token,
            "product_code": BLUESPARK_PRODUCT_CODE,
        },
    )

def get_current_user(token: str = Depends(oauth2_scheme), db: Session = Depends(get_db)):
    claims = introspect_central_token(token)
    if not claims.get("active"):
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Sessao invalida.")
    return sync_local_user_from_claims(db, claims)

@router.get("/me")
def me(current_user = Depends(get_current_user)):
    return current_user

O que um app não deve mais fazer

Tirar do app consumidor

  • Cadastro de senha local
  • Google login próprio
  • Recuperação de senha local
  • Emissão própria de JWT do utilizador global
  • Refresh token do utilizador global

Manter no app consumidor

  • Perfis locais e dados do domínio
  • Permissões do produto
  • Autorização de negócio
  • Objetos locais ligados ao user
  • Admin técnico interno, se separado

Checklist de integração

Cadastrar o product_code do app na BlueSpark.
Configurar BLUESPARK_ACCOUNTS_URL e BLUESPARK_PRODUCT_CODE.
Garantir que a tabela local tem central_user_id.
Migrar login, Google auth, recovery e refresh para a BlueSpark.
Fazer introspecção do token no backend do app.
Criar ou sincronizar o utilizador local automaticamente.

Leitura do problema atual no SkyVenda

Se a BlueSpark autentica e o SkyVenda não, o problema normalmente está em uma destas camadas:

  • product_code errado ou ausente no pedido do SkyVenda.
  • Token emitido para BlueSpark, mas não aceito pelo backend SkyVenda.
  • SkyVenda ainda tenta ler JWT antigo localmente.
  • Falta de criação ou vínculo do user local com central_user_id.
  • Frontend do SkyVenda ainda chama rotas legadas em vez das rotas centralizadas.
